ok, why not use a medallion structure

You can use a medallion structure here — and you probably should. The key is: don’t treat “medallion” as “copy tables three times.” Treat it as contracts + responsibilities per layer.

With Supabase → Snowflake → Hex, medallion maps cleanly:

Bronze = RAW (1:1 ingestion, auditability)

Silver = STG/CORE (cleaned, conformed entities)

Gold = MARTS/METRICS (facts, aggregates, semantic views)

The reason I didn’t lead with the word “medallion” is that in Snowflake + dbt land people often call it raw/stage/core/marts, but it’s the same idea.

Why medallion works well for your stack
1) Debuggability + trust

When Growth says “this number looks wrong,” medallion lets you answer:

Is it wrong in bronze (source/ingest)?

wrong in silver (logic/conformance)?

or wrong in gold (aggregation/metric definition)?

That’s how you build confidence fast.

2) Separation of concerns

Bronze is owned by data engineering reliability

Silver is owned by data modeling correctness

Gold is owned by business metric definitions

That separation is exactly what a founding data hire needs.

3) Hex self-serve becomes safe

Hex should mostly query Gold (and sometimes Silver).
If Hex hits Bronze directly, you’ll get:

inconsistent joins

duplicated logic

“two versions of revenue”

The gotcha: medallion can go wrong in Snowflake

Here’s why some teams avoid it:

A) “Bronze” becomes a messy replica forever

If Bronze is just “copy tables over,” you end up with:

a swamp of raw tables

no clear contracts

endless ad-hoc Hex joins

Fix: Bronze must have strict rules:

immutable or CDC-based

minimal transforms

consistent metadata (_loaded_at, _source, _deleted)

B) Silver becomes “whatever”

If Silver is not well-defined, you recreate chaos one layer up.

Fix: define Silver as:

staging (type casting, normalization, dedupe)

conformed core entities (1 row per entity, stable keys)

C) Gold becomes “dashboard spaghetti”

If Gold is just a bunch of random aggregates, you’ll have metric drift.

Fix: Gold should be:

facts at stable grains (session/request/billing_event)

dimensions (org/user/plan/date)

semantic metric views (canonical MRR, DAU, etc.)

when you say replicating - do you mean like copying the database 

Yes — but specifically copying it continuously and incrementally, not just a one-time dump.

In data engineering, when we say replicating Supabase → Snowflake, we mean:

Maintaining a continuously updated copy of your Supabase (Postgres) tables inside Snowflake.

Not a static export. A live mirror.

There are 3 levels of “copying the database”
Level 1 — Bad: one-time dump

Example:

Export Postgres tables to CSV

Upload CSV to Snowflake

Done

Problem: immediately stale.

This is useless for analytics beyond debugging.

Level 2 — OK: periodic full copy

Example:

Every hour:

SELECT * FROM users;
SELECT * FROM sessions;
SELECT * FROM subscriptions;

Load into Snowflake.

Problem:

expensive

slow

inefficient at scale

Still works early-stage.

Level 3 — Correct: incremental replication (CDC)

This is what you want.

Instead of copying everything, you copy:

new rows

updated rows

deleted rows

Only changes.

This is called Change Data Capture (CDC).

How CDC replication actually works technically

Postgres (Supabase) has a transaction log called the WAL (Write Ahead Log).

Every change is recorded:

insert user

update session

delete subscription

Replication tools read this log and apply the same changes in Snowflake.

So Snowflake becomes a near-real-time replica.

Example

Supabase table:

users

Supabase:

id	email	created_at
1	a@x.com
	Jan 1
2	b@x.com
	Jan 2

Snowflake replica:

Same table appears in Snowflake:

BRONZE_SUPABASE.users

Same rows.

Then a new user signs up:

INSERT INTO users VALUES (3, 'c@x.com', NOW())

Replication tool detects it and inserts into Snowflake automatically.

No manual export needed.

Why replication is critical

You never query Supabase directly for analytics because:

Supabase is optimized for:

fast writes

application performance

Not for:

large scans

heavy joins

analytics queries

Snowflake is optimized for analytics.

Replication bridges the gap.

In your Browserbase case, the architecture is likely:
Supabase (Postgres)
    ↓ replication
Snowflake (warehouse)
    ↓ models/views
Hex (queries, dashboards)

Supabase powers the product.

Snowflake powers analytics.

What replication actually creates in Snowflake

Example schema:

BRONZE_SUPABASE.users

BRONZE_SUPABASE.organizations

Old ETL:

how do i make a replica

“Make a replica” (Supabase/Postgres → Snowflake) usually means set up a connector that continuously copies tables + changes (CDC) into Snowflake.

There are three practical ways, in order of “fastest to production”:

Option A (fastest): Fivetran Postgres → Snowflake (CDC)

When to use: you want the replica working today with minimal ops.

Steps

Create a read/replication user in Supabase (Postgres)

Needs read access to the schemas/tables you want.

Needs permissions for logical replication (CDC) if you use it.

Enable CDC / logical replication

Fivetran will guide this, but conceptually you need:

A publication (which tables to stream changes for)

A replication slot (cursor in the WAL stream)

Typical SQL pattern (exact permissions depend on Supabase setup):

-- choose tables (or ALL TABLES if you're comfortable)
CREATE PUBLICATION fivetran_pub FOR ALL TABLES;

Network access

Your Supabase instance must allow inbound connections from Fivetran.

Usually this is “allowlisted IPs” + SSL.

Configure destination in Snowflake

Pick database/schema like BRONZE_SUPABASE.

Fivetran will create and maintain the tables.

Verify

Check that inserts/updates in Supabase show up in Snowflake within minutes.

What you get in Snowflake: a near-1:1 copy of tables (plus metadata columns like _fivetran_synced, etc.).

Option B (cheaper, more DIY): Airbyte Postgres → Snowflake

When to use: you want to save on SaaS cost and can handle a bit of ops.

Steps

Deploy Airbyte (cloud or self-hosted).

Add Source: Postgres (your Supabase connection string).

Choose replication mode:

CDC (best) if supported/allowed

or incremental by updated_at (good enough early, but deletes are tricky)

Add Destination: Snowflake (warehouse/db/schema).

Turn on auto schema evolution (optional) and set sync frequency.

Run initial sync; then verify incremental updates.

Key caveat: if you can’t do true CDC, you must standardize updated_at everywhere + decide how to handle deletes (soft delete flag or periodic full refresh).

Option C (most control): Debezium CDC → Snowflake (custom)

When to use: you need full control, very high volume, or strict requirements.

Typical architecture

Supabase Postgres WAL → Debezium (reads changes)

Debezium → Kafka/Redpanda

Kafka → Snowflake (Snowflake Kafka connector) or write files → Snowpipe

This is powerful but heavier to run and maintain.

The Postgres pieces you’re actually creating

To do a true “replica,” you’re enabling logical replication (CDC). Conceptually:

Publication = “which tables’ changes should be emitted”

Replication slot = “where am I in the WAL stream”

Most managed tools create/maintain the slot for you; you usually create the publication.

Two “gotchas” that bite people

RLS / permissions in Supabase
Replication user needs access that bypasses RLS for analytics; otherwise you replicate partial data.

Deletes

CDC handles deletes correctly.

“Incremental by updated_at” does not unless you do soft deletes or periodic full refresh.
